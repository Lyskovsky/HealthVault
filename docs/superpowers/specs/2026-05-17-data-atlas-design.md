# Data Atlas — спецификация

**Дата:** 2026-05-17
**Автор:** Александр Лысковский + Claude
**Контекст:** первая сессия архитектурного брейнсторма по итогам Granola «Botkin 16.05». Боль: данные раскиданы между Mac / Apple Cloud / Hetzner / Google Drive, нет единой карты. Без аудита текущего состояния любая миграция (server-side sync, NanoCloud, MCP-сервер) идёт вслепую.

---

## Цель

Получить один живой md-документ, который отвечает на два вопроса:

1. **«Где у меня данные?»** — для каждого потока: источник, транспорт, физическое хранилище, частота обновления, кто читает.
2. **«Где у мульти-юзера дыры?»** — те же потоки в разрезе 4 активных пользователей (Александр / Ника / Андрей / Олег).

## Не-цели

- Не план миграции. Атлас — только описание текущего состояния.
- Не история (как было раньше). Только «здесь и сейчас».
- Не учебник для нового разработчика — это документ для владельца и техлида.

## Артефакт

- **Файл:** `docs/DATA_ATLAS.md` (top-level в `docs/`, не закопан в `superpowers/`)
- **Формат:** Markdown, читается в Obsidian / Notion / GitHub
- **Обновление:** дата в шапке документа (не в имени файла); статус — «живой», обновляется при изменениях архитектуры
- **Длина:** ~250-400 строк (одна таблица + 5 коротких разделов)

## Структура документа

```
1. Header (10 строк)
   - Что это, дата обновления, как поддерживать
2. Executive summary (1 абзац, 5-7 предложений)
   - Главные находки + 3 ключевых красных флага
3. Главная таблица потоков (stream-first, 16+ строк)
   - Колонки: Поток | Источник | Транспорт | Хранится | Обновл. | Читают
                | Alex | Ника | Андрей | Олег
4. Physical Storage Map (1 страница, ~50 строк)
   - Mac (~/.../Botkin/data/) — что лежит, размер
   - Hetzner Postgres (/opt/healthvault) — какие таблицы, объём
   - Google Drive (~/FamilyHealth/) — структура папок, JSON KBs
   - Apple Cloud (HAE webhook) — proxy без persistent storage
   - Garmin Cloud / Netatmo Cloud / Zepp Cloud — внешние, тянем по API
5. Pipeline diagrams (3-5 mermaid)
   - Apple Health → HAE → Postgres
   - Medical PDF → Claude extract → knowledge_base.json
   - Garmin → push_garmin_to_db.py → Postgres
6. Gaps & Risks (главная ценность аудита, ~30 строк)
   - Минимум 3 архитектурных вывода
7. Methodology (приложение, ~20 строк)
   - Команды/SQL, которыми собирался — чтобы повторить было реально
```

## Колонки главной таблицы

| Колонка | Что в ней | Пример |
|---|---|---|
| **Поток** | Короткое имя | Питание, Сон, Шаги, Давление |
| **Источник** | Устройство / человек | сообщения боту, Apple Watch, Garmin, Omron |
| **Транспорт** | Как доходит до хранения | HAE webhook, API pull, ручной ввод |
| **Хранится** | Конкретное физическое место | `Postgres.nutrition_log`, `data/garmin/daily-summary/`, `~/FamilyHealth/{user}/knowledge_base.json` |
| **Обновл.** | Частота / триггер | при записи / nightly / manual / weekly |
| **Читают** | Кто потребитель | бот, dashboard, KB-импорт, MCP (будущее) |
| **Alex** | Статус для Александра | ✅ работает / ⚠️ частично / — нет |
| **Ника** | Статус для Ники | то же |
| **Андрей** | Статус для Андрея | то же |
| **Олег** | Статус для Олега | то же |

## Методология сбора данных

1. **Inventory из существующих документов** — `CLAUDE.md` уже содержит таблицу источников; начать с неё, не изобретать заново.
2. **Server SQL** — `\dt` + per-table `SELECT COUNT(*), MAX(date), COUNT(DISTINCT user_id) FROM <table>`.
3. **File scan локально** — `du -sh data/*/`, `ls -la data/garmin/daily-summary/ | tail`.
4. **Code trace** — для каждого скрипта в `scripts/import/` понять что читает / куда пишет.
5. **Config-аудит** — где per-user токены/ключи (`.env`, `users.garmin_email`, `users.health_token`).
6. **Cloud-аудит** — внешние сервисы (Garmin Connect, Netatmo, Zepp) — что мы тянем и куда складываем.

## Критерии готовности

- Каждая строка главной таблицы заполнена для всех 4 пользователей
- Каждый Physical Storage location перечислен с размером и кратким описанием
- Раздел **Gaps & Risks** содержит **минимум 3 архитектурных вывода** (если меньше — аудит был поверхностный)
- Mermaid-диаграмма Apple Health pipeline корректно рендерится в GitHub-preview
- В Executive Summary не больше 7 предложений (иначе кто-то напишет ТЗ вместо отчёта)

## Трудозатраты

**~4-6 часов** фокусированной работы (полный рабочий блок). Не «между делом за вечер».

Этапы:
1. SQL-запросы и file-сканы (1ч)
2. Заполнение таблицы (1.5ч)
3. Physical storage map (1ч)
4. Pipeline diagrams (1ч)
5. Gaps & Risks (раздел с инсайтами — самое ценное, 1ч)
6. Self-review + правки (30 мин)

## Что НЕ делаем в этой итерации

- Не пишем `audit_data.sh` (это была опция B, не выбрана)
- Не делаем интерактивный мониторинг (опция C, не выбрана)
- Не предлагаем решений архитектурных проблем (это уже следующая сессия — спека по миграции)
- Не описываем данные за 11 лет — только текущее состояние pipeline
